Step 3 — Enable Netlify Identity

Identity handles user login, invites, and JWT tokens that protect your dashboard.

1
Go to your site dashboard → Integrations → Identity
Click Enable Identity. This activates the invite-only user system.
2
Set registration to "Invite only"
Identity → Settings → Registration: select Invite only. This prevents self-signup — only users you invite can register.
3
Enable Git Gateway (optional, for future CMS use)
Identity → Services → Enable Git Gateway. Not strictly required but good practice.
⚠️
Important: After enabling Identity, go to Identity → Settings → External providers. Keep everything off — you only want email-based invite login, not Google/GitHub OAuth.

Step 4 — Set Environment Variables

Go to Site configuration → Environment variables → Add variable and add all of the following:

Variable Name Value / Description Required
ADMIN_EMAIL Your email address — the one you'll use to log in as admin. Must match your Netlify Identity account exactly. Required
VALID_LICENSE_KEYS Comma-separated list of valid license keys. Example: KEY-ALPHA-2024,KEY-BETA-2024,KEY-GAMMA-2024. You give these to users who should have access. Required
KITE_API_KEY Your Zerodha Kite Connect API key (from kite.trade developer console). Required
KITE_ACCESS_TOKEN Your Zerodha access token — regenerated daily. Set once manually; afterward the admin dashboard can update it automatically each morning. Required
NETLIFY_SITE_ID Your Netlify site ID — found in Site configuration → General → Site ID (looks like a UUID: abc123de-f456-...). For auto-token
NETLIFY_API_TOKEN A Netlify personal access token. Create one at app.netlify.com → User settings → Applications → Personal access tokens. Needed to let the admin dashboard save the daily Kite token automatically. For auto-token
💡
Tip — Token scope: When creating the Netlify personal access token, give it a descriptive name like "RSI Trader Token Updater". No special scope selection is needed on the free plan — the token gets full API access.

Example VALID_LICENSE_KEYS format

ENV VAR VALUE
TRADE-ALPHA-0001,TRADE-BETA-0002,TRADE-GAMMA-0003

You hand each user one of these keys. They enter it on the login page to "activate" their copy before logging in.

Managing License Keys

License keys are stored in the VALID_LICENSE_KEYS environment variable as a comma-separated list. To add or revoke a key:

1
Edit the env var in Netlify dashboard
Site configuration → Environment variables → click VALID_LICENSE_KEYS → Edit → update the list → Save.
2
Trigger a redeploy
Functions read env vars at cold-start. A redeploy (or waiting ~a few minutes for the next warm-up) is enough. Removing a key from the list immediately rejects it on next use.
⚠️
License keys are stored in localStorage on the user's browser after activation. Revoking a key prevents new activations but won't immediately log out existing sessions — they'd need to clear their browser data or you'd need to revoke their Identity account.

Zerodha / Kite API Setup

One-time setup

1
Create a Kite Connect app at kite.trade
Log in → Developer console → Create App. Set the redirect URL to your site: https://your-site.netlify.app/. Note your API Key and API Secret.
2
Get your first access token
You need to complete the OAuth login flow once manually to get the first token. See the code block below.
3
Add KITE_API_KEY and KITE_ACCESS_TOKEN to Netlify env vars

Getting the initial access token (manual, one time)

STEP 1 — open this URL in your browser (replace with your API key)
https://kite.zerodha.com/connect/login?api_key=YOUR_API_KEY&v=3
STEP 2 — after login, you're redirected to your site with a request_token in the URL
https://your-site.netlify.app/?request_token=XXXXXX&action=login&status=success
# Copy the request_token value from the URL
STEP 3 — exchange it for an access token (run in Node.js or any REST client)
# Using curl:
curl -X POST https://api.kite.trade/session/token \
  -H "X-Kite-Version: 3" \
  -d "api_key=YOUR_API_KEY" \
  -d "request_token=YOUR_REQUEST_TOKEN" \
  -d "checksum=SHA256(api_key+request_token+api_secret)"

# The response contains "access_token" — copy that value
# and paste it into KITE_ACCESS_TOKEN in Netlify env vars
💡
Daily token flow: After the first manual setup, you can use the admin dashboard's "Connect Zerodha" panel each morning to log in via Kite and update the token automatically. The save-token.js function handles persisting it to your Netlify env vars so all users pick it up.

Daily Token Refresh

Zerodha access tokens expire every day at midnight. Each trading day, you need to refresh the token:

1
Visit your site and log in as admin
2
Go to the Zerodha Connection panel in the dashboard
The admin-only "Platform API" section appears at the top of the sidebar when logged in as admin.
3
Enter your API key + new access token → click Connect
The dashboard calls save-token.js which updates KITE_ACCESS_TOKEN in your Netlify env vars via the Netlify API. All users' API calls use the new token instantly.
⚠️
Note: The token update only persists if you have NETLIFY_API_TOKEN and NETLIFY_SITE_ID set. Without these, the token works only for the current session and resets on next deploy.

Troubleshooting

License key validation fails

⚠️
Check that VALID_LICENSE_KEYS is set and contains your key. Keys are compared case-insensitively (uppercased). Check Functions logs for the exact rejection message.

Dashboard shows blank / redirects to login

⚠️
The netlify.toml redirects /dashboard.html to /index.html when no Identity role is present. Make sure you're logged in (Identity JWT token must be active in the browser). Try clearing cookies and logging in again.

Kite API returns 401/403

⚠️
Your KITE_ACCESS_TOKEN has expired (they last only ~8 hours / until midnight). Re-connect via the admin panel or manually update the env var and redeploy.

Functions not found (404)

⚠️
Verify that netlify.toml has functions = "functions" and your JS files are in a folder literally named functions/ (not netlify/functions/). The path in the TOML is relative to the repo root.

Identity widget not appearing

⚠️
Make sure Netlify Identity is enabled for your site (Integrations → Identity → Enable). The CDN script in index.html only works on Netlify-hosted sites where Identity is active.

Admin panel not visible after login

⚠️
The is-admin.js function compares your logged-in email against ADMIN_EMAIL exactly. Check for trailing spaces, and make sure you're logged in with the exact same email. Case-insensitive comparison is applied.

Security Notes

🔒
API keys never reach the browser
KITE_API_KEY and KITE_ACCESS_TOKEN live only in Netlify env vars and are used exclusively inside kite-proxy.js (a server-side function). Regular users can never see them.
🔒
Endpoint whitelist in kite-proxy.js
The proxy only forwards requests to a hardcoded whitelist of Kite endpoints (/quote, /orders/regular, /positions, etc.). Requests to any other endpoint are rejected with 403.
🔒
Identity JWT is verified server-side
Every serverless function checks event.clientContext.user — this is populated by Netlify from a verified JWT, not from anything the browser sends. It cannot be spoofed by client-side code.
⚡
Production hardening recommendation
The validate-license.js function returns the license key back to the client and stores it in localStorage unencrypted. For stricter control, consider replacing this with a server-signed JWT (HMAC with a secret) so the client token cannot be tampered with.
🚨
Never commit secrets to Git. Your KITE_API_KEY, NETLIFY_API_TOKEN, license keys, and all credentials must only be set in Netlify's Environment Variables UI — never hardcoded in any file in your repo.
